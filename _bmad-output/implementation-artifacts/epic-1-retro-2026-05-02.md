# Epic 1 Retrospective - Secure Runtime Foundation

Date: 2026-05-02
Epic: 1
Facilitator: John (Product Manager)
Participants: Product, Developer, QA, Security

## Epic Review Summary

- Epic status: complete
- Stories complete: 3/3
- Story keys:
  - 1-1-startup-validation-and-setup-gating
  - 1-2-webhook-verification-and-signature-enforcement
  - 1-3-correlation-logging-and-observability-baseline
- Retrospective scope: security foundation correctness, configuration gating accuracy, observability contract stability, and carry-forward lessons for downstream epics

---

## What Went Well

1. **Secure startup gating was structurally sound from the start.** The app-factory pattern correctly separated startup validation from request handling; the webhook POST short-circuit via `config_invalid` (503) ensured traffic never reached business logic with an incomplete configuration.

2. **Centralized sanitization delivered durable value.** `SafeObservabilityFilter` in `app/services/observability.py` became a stable redaction primitive that all downstream epics (2, 3, 5, 6) relied upon without re-implementing. One point of ownership meant improvements in Epic 5 and 6 accumulated safely.

3. **Correlation ID propagation was implemented correctly at the app-factory layer.** Binding correlation IDs in `before_request`/`after_request`/`teardown_request` hooks meant every downstream story could assume traceability without custom per-route plumbing.

4. **HMAC signature enforcement was centralized in a decorator.** `@signature_required` became the single admission gate for all inbound POST traffic, ensuring no route could accidentally bypass validation.

5. **Test suite coverage expanded early and held.** Story 1.1 grew from 22 to 31 passing tests after the `None`-config false-positive fix; Story 1.3 reached 44 tests across the 1.1/1.2/1.3 suite with full provider isolation. This gave subsequent epics a reliable regression baseline.

6. **`/health` and `/metrics` contracts were published as stable public endpoints.** Downstream dashboard and release-gate work in Epics 3, 4, and 6 consumed these without requiring renegotiation.

---

## What Did Not Go Well

1. **`str(None)` truthiness caused a false-positive config readiness bug.** The initial `is_config_value_set` implementation passed `str(None)` through as a truthy string, reporting required keys as present when they were actually unset. This required a targeted fix and re-expansion of the test suite.

2. **`SECRET_KEY` had a hardcoded fallback that allowed session forgery during setup.** A review finding identified that `app/config.py`'s `load_configurations` used a static fallback secret, which created a session integrity gap for operators in an incomplete-setup state. This was patched but was a security regression that should have been caught before dev completion.

3. **Provider mode isolation was not enforced in tests at the start of Story 1.3.** Tests in `WebhookVerifyTests`, `WebhookSignatureEnforcementTests`, `SetupRouteTests`, and `EvolutionWebhookTests` did not use `patch.dict(os.environ, ..., clear=True)`. Because `.env` loads Evolution keys globally, Meta-path tests were failing intermittently in environments where both provider modes were configured. This required a systematic fix across all test classes.

4. **Operator metrics route was split across two blueprints without being reconciled.** `/metrics` (public JSON, `views.py`) and `/operator/metrics` (HTML dashboard, `views_dashboard.py`) diverged during implementation and were not explicitly documented as separate surfaces until Epic 4 review surfaced the inconsistency.

5. **Observability seam re-entry was not guarded against handler accumulation.** When the app factory re-ran in test contexts, logging handlers could accumulate, causing repeated filter application. This was addressed as part of Epic 5 hardening but should have been caught in Epic 1.

---

## Root Causes

1. **No explicit boundary check for `None` vs blank string in config validation helpers.** The shared `is_config_value_set` helper was implemented with implicit truthiness rather than explicit type-and-value checks, allowing edge cases through.

2. **Security review happened at completion rather than during story scoping.** The `SECRET_KEY` fallback gap was a design-phase oversight — it should have been a stated acceptance criterion for Story 1.1, not a post-implementation finding.

3. **Provider-mode isolation was treated as an implementation detail rather than a test fixture requirement.** Tests were written without acknowledging that `.env` file loading could inject global environment state, making provider isolation an invisible dependency.

4. **Route ownership across public/operator surfaces was not explicitly mapped at Epic 1 scope.** The `/metrics` vs `/operator/metrics` split emerged organically, without a canonical route inventory that would have surfaced it earlier.

5. **Logging handler lifecycle in test isolation was not a defined acceptance criterion.** The test infrastructure for observability was treated as incidental rather than as a first-class concern during story scoping.

---

## Concrete Process Improvements Identified

1. Add explicit `None` and empty-string validation to all config-check helpers as a default requirement, not an optional hardening step.
2. Include a dedicated security acceptance criterion in every story that touches session state, secret handling, or authentication pathways.
3. Standardize test fixtures for provider isolation using `patch.dict(os.environ, ..., clear=True)` as a non-negotiable baseline for any test involving `WHATSAPP_PROVIDER`.
4. Publish a canonical route inventory (public vs operator surfaces) at the start of each epic that adds new routes, and validate it in tests.
5. Make logging handler accumulation protection a defined seam requirement when observability infrastructure is first implemented, not a deferred hardening fix.

## Carry-Forward Actions for Future Work

| Action | Owner | Success Criteria |
| --- | --- | --- |
| Reuse `is_config_value_set` from `app/config.py` in all future config validation helpers to prevent None-path regressions | Developer | No new config helpers use bare truthiness checks |
| Treat `SECRET_KEY` and session-related config as first-class security acceptance criteria in any future story touching setup or authentication | Product / Developer | Security AC present in story acceptance criteria before dev starts |
| Enforce `patch.dict(os.environ, ..., clear=True)` in all tests involving provider or environment-specific config | Developer | No test failures attributable to cross-environment state bleed |
| Maintain explicit route inventory in `docs/` or architecture artifacts updated alongside any blueprint that adds operator vs public routes | Architect / Developer | Route inventory always matches implemented routes before epic closure |
| Document logging handler guard requirement (max-one-handler-per-logger pattern) in project architecture notes | Developer | No test or production instance accumulates duplicate log handlers |

---

## Continuity Notes

- Epic 1 established the security and observability foundation that every subsequent epic depended on. Its seams held without regression for Epics 2–6.
- The key fragilities surfaced in review (None-config bug, SECRET_KEY fallback, provider isolation) were all addressed before they propagated; the cost was additional fix cycles that could have been prevented by more explicit acceptance criteria at scoping time.
- The most durable decision Epic 1 made was centralizing sanitization in a single filter — this avoided redaction re-implementations across six epics and should be the model for any new cross-cutting concern.
