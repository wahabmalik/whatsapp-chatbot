## What Went Well

1. Security and integrity hardening landed across operator-critical paths:
   - CSRF enforcement on dashboard mutation routes.
   - serialized + atomic `.env` updates.
2. Runtime reliability controls improved:
   - explicit `WHATSAPP_DEFER_RETRIES` and worker configurability.
   - clearer background delivery ownership in a dedicated service.
3. Observability baseline became safer and cleaner:
   - correlation ID cap and idempotent filter ownership.
   - sanitization support expanded to additional container types.
4. Setup and operator UX precision improved:
   - setup/access flows and runtime update behavior are more deterministic.
   - live OpenAI client refresh path removed restart ambiguity for key updates.
5. Deferred-work backlog for this hardening slice was actively burned down, leaving only non-actionable theoretical items.

## What Did Not Go Well

1. Some behavior contracts remain partially implicit in code rather than fully codified in docs/tests (for example exact fallback retry semantics under mixed failure sequences).
2. Setup step-state semantics are still easy to drift if UI and controller logic evolve separately.
3. A few security/observability edge patterns need stronger regression locking to avoid future leakage regressions.

## Root Causes

1. Hardening touched multiple layers (config, routes, async delivery, observability), which increases coupling risk when acceptance checks focus mostly on happy paths.
2. Operationally important edge-cases (retry/fallback permutations, token pattern variance) are underrepresented compared to primary flow tests.
3. UI-state correctness depends on explicit state modeling; without shared state definitions, templates and route helpers can diverge.

## Lessons Learned

1. Hardening is most effective when policy and test assertions are explicit and symmetric.
2. Security filters should assume evolving token formats and sanitize by robust patterns, not narrow historical assumptions.
3. Async reliability features need end-to-end observability and deterministic replayable tests to keep operator trust high.

## Next Epic Preparation

1. Convert remaining implicit retry/escalation policies into explicit contract tests.
2. Add a small cross-layer conformance check for setup step progression (controller output vs rendered ARIA state).
3. Expand log-sanitization tests to future-proof secret pattern handling.
4. Add one reliability stress test around deferred delivery completion logging and operator artifact queueing.
