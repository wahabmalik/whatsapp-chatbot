The file listing shows duplicate entries:
- `7-10-config-truthiness-boundary-contract-tests.md` and `7-10-test-coverage-map-and-baseline-test-count.md`
- `7-11-secret-key-hardcoded-fallback-absence-test.md` and `7-11-test-coverage-map-and-baseline-test-count.md`

**Root Cause:** Sprint-status.yaml shows 7-12-test-strategy-index correctly, but the actual artifact is named 7-12-test-coverage-map. There was a naming drift between planning and execution.

**Impact:** Minor — the artifact exists and is correct, but the naming confusion could slow down future developers searching the file system.

**Learning:** Validate artifact filenames against sprint-status.yaml during story closure, not at epic retro.

### 2. **No Explicit Baseline Confirmation at Epic Open**

Epic 7 opened with an assumption: "All Epic 6 tests are passing as baseline." There was no recorded confirmation run at epic open to establish a clean starting point.

**Impact:** If a test failure had appeared mid-epic, it would be ambiguous whether it was a regression from Epic 7 work or a carry-forward from Epic 6.

**Evidence of Impact:** Minimal, since the epic closed clean, but the pattern is risky for future carries.

### 3. **Some Story Boundaries Required Clarification During Execution**

Stories like 7-4 and 7-5 required real-time discussion during implementation:
- "Does this test touch production code?" (No, keep it test-only)
- "Which existing test file should this extend?" (Decided during dev)

**Root Cause:** Carry-forward stories drafted from retrospective action items often lack the detailed design clarity of initially-planned stories.

**Impact:** Low — stories still delivered on time, but dev had to make scope decisions that ideally would have been pre-documented.

### 4. **Contract Test File Distribution Across 3+ Files Creates Traceability Tax**

Tests for Story 7-6 (retry/fallback) appear in multiple files:
- `test_retry_escalation_contract.py`
- `test_reliability.py`
- `test_release_gates.py`

Tracing a single behavioral contract across 3+ files is cognitively expensive during code review.

**Root Cause:** Tests were added to the "most appropriate existing file" without maintaining a top-level strategy index until Story 7-12.

**Impact:** Manageable now that Story 7-12 created the coverage map, but creates ongoing maintenance friction.

### 5. **One Pre-Existing Test Failure Remains Unaddressed**

The test `test_story_3_3.py::OperatorLogsViewTests::test_logs_filter_status_error` failed before Epic 7 opened and remains failing at epic close.

**Evidence:**
- Flagged as pre-existing in Epic 7 context statement
- Not addressed because it was out-of-scope for Epic 7's carry-forward focus
- But Epic 7 didn't create a formal carry-forward action to close it

**Risk:** This test will silently fail in all future runs unless explicitly assigned to a future epic.

## Root Causes and Patterns

### 1. **Carry-Forward Sprints Need Explicit "Out of Scope" Boundaries**

Epic 7 was effective because it had a clear scope: "Close Epic 6 action items + contract tests from earlier epics." But there was no documented boundary. Future developers need to know: "We're NOT opening new bugs, we're NOT refactoring production code beyond docs, we're NOT adding new features."

### 2. **Test File Distribution Follows Historical Proximity, Not Strategy**

When a test gets added, developers naturally ask "Which file has similar tests?" rather than "Where does this logically belong in a test strategy?" Over time, related tests scatter across multiple files. Story 7-12's coverage map partially solves this, but a top-level test strategy document would prevent future drift.

### 3. **Documentation Artifacts Require Earlier Discipline Than Production Code**

Documentation-only stories (like 7-8 and 7-12) often have ambiguous done-ness criteria. "The doc exists" is not the same as "the doc is discoverable" or "the doc is accurate." Story 7-12 required real-time discussion of what goes in the coverage map; a template would have clarified expectations upfront.

### 4. **Baseline Metrics Must Be Captured at Epic Boundaries**

Without a recorded baseline, it's impossible to prove the epic achieved its goals ("we added 28 tests") vs. to measure regression risk. Story 7-12 was supposed to capture this but the baseline capture was deferred. Future epics should capture baseline metrics at day 1 and day 12 automatically.

## Architectural Insights Gained

### 1. **Sprint Integrity Can Be Automated**

The YAML duplicate-key problem (Epic 6) was a manual process failure. Adding a CI gate (Story 7-1) transforms it from a risk into a non-issue. This pattern — "turn manual discipline into automated guardrails" — is worth repeating.

### 2. **Documentation Becomes Safer When Testable**

Story 7-2 treats documentation as data: parse it, validate it against reality. This is the inverse of traditional "write docs, hope they stay accurate." Operational docs that are machine-checked will degrade predictably when they drift.

### 3. **Architecture Decisions Need Visible Anchors**

Story 7-8's comment block in code is more discoverable than a separate architecture document because developers see it during code review. The pattern: "Put decision logic next to implementation as an unavoidable comment."

### 4. **Boundary Condition Testing Requires Explicit Intention**

Story 7-10 tests values (`None`, `""`, `"0"`) that would fail silently if the config validation logic subtly changed. Python's truthiness rules are notoriously surprising. Explicit parametric tests make these contracts visible.

## Lessons for Future Epic Planning

### ✅ What Worked Well in This Retrospective Carry-Forward Pattern

1. **Explicit carry-forward action items from previous retro** — Eliminated ambiguity about what "carry-forward" means
2. **Documentation-first approach** — Tests that verify docs stay aligned with code are high-value
3. **Zero production code changes except necessary docs** — Keeps scope tight and risk low
4. **Baseline-to-baseline metrics** — Proves the epic achieved measurable goals
5. **Story-level transparency** — Each story file explains the "why" for future developers

### ⚠️ What to Improve Next Time

1. **Capture baseline metrics at epic open, not close** — The "Epic 7 open" line in the table above is an assumption
2. **Pre-populate story templates with explicit non-scope boundaries** — Reduces mid-execution clarification questions
3. **Assign carry-forward carry-forward actions to Epic N+2, not later** — "Future work" action items accumulate debt
4. **Run a clean pytest baseline run at epic start with results recorded** — Provide proof that Epic N-1 left the codebase in good shape
