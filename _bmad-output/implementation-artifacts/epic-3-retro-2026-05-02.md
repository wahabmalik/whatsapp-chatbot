Epic: 3

Participants: Product, Developer, QA, UX

- Epic status: complete
- Stories complete: 3/3
- Story keys:

  - 3-3-conversation-context-and-operator-activity-views
- Retrospective scope: operator control-plane correctness, escalation signal determinism, setup wizard UX fidelity, dashboard accessibility, and lessons for release readiness and hardening epics

---

## What Went Well

1. **Atomic agent selection persistence was correctly implemented from day one.** `tempfile.NamedTemporaryFile` + `os.replace` ensured concurrent writes could never produce a corrupt `data/agent_selection.json`. The stale-selection auto-repair logic also meant runtime kept a valid agent even after manual file edits.

2. **Agent discovery merged manifest and skills sources without duplication.** Deterministic sorting by agent code kept the UI rendering stable across discovery runs, avoiding ordering flicker for operators.

3. **Operator route guards and safe redirect logic were centralized.** The `_get_safe_redirect_target` helper rejecting absolute and protocol-relative URLs prevented open-redirect exposure in operator navigation paths from the start.

4. **Escalation taxonomy was made deterministic and extensible.** Explicit reason codes (`low_confidence`, `escalation_keyword`, `outbound_fallback_failure`) gave downstream log surfacing and operator review queue a stable vocabulary — no ambiguous free-text reasons that would require parsing.

5. **`MessageLogBuffer` (FIFO, max 100 entries) was lightweight and required no infrastructure addition.** In-memory design kept the dashboard log surface dependency-free and testable with simple fixtures.

6. **Conversation context window (last 5 per user, timeout-reset) was implemented as a dedicated service** with a narrow interface (`append_message`, `get_context`, `reset_context`, `clear`). This bounded the FR9 compliance surface to one owner and made the reset invariants explicitly testable.

7. **Masked phone numbers with explicit opt-in reveal preserved privacy by default** across the logs view — a correct default that would have been difficult to retrofit later.

---

## What Did Not Go Well

1. **Story 3.2 reached review state rather than done at epic close.** The setup wizard escalation workflow had remaining gaps (aria-live semantics, setup completion redirect preserving operator mode, escalation artifact queue formalization) that required continued work after the epic story list was otherwise complete. This introduced a temporary quality inconsistency within Epic 3.

2. **Agent registry had a runtime `NameError` in `list_bmad_agents`.** A `for agent in agents` reference without prior assignment caused the agent listing route to fail at runtime. This was a functional regression that should have been caught before story completion and reflects insufficient integration testing of the discovery path.

3. **Template used a hardcoded path instead of `url_for()` for agent save POST.** `app/templates/agents.html` posted to a hardcoded path rather than `url_for('webhook.set_agent_selection')`, which breaks in mounted/subdirectory deployments — a UX correctness issue invisible in local development.

4. **Mobile navigation was missing from the operator dashboard specifically.** `app/templates/base.html` omitted bottom navigation when `page_key == 'dashboard'`, contradicting UX requirements. This inconsistency existed in the codebase despite being in scope for Epic 3.

5. **Inline expand/collapse for log rows was not implemented.** The logs view rendered always-visible secondary detail rows rather than true collapsible per-row states specified in the UX design, leaving an accessibility and readability gap.

6. **Conversation context memory was not a dedicated app service prior to Story 3.3.** OpenAI thread IDs were stored in a shelf without FR9 rolling-window or reset enforcement, meaning per-user context limits were not honored before this story closed the gap.

---

## Root Causes

1. **Story 3.2 UX and accessibility acceptance criteria were detailed but verification was incomplete at review.** Aria semantics and redirect preservation were each correctible in isolation, but treating them as granular sub-tasks without an integration test for the full setup wizard lifecycle allowed them to slip.

2. **Agent registry path was not covered by a runtime smoke test.** The `NameError` in `list_bmad_agents` was a trivial but impactful code defect — one integration test exercising the agents route against a live app fixture would have caught it before story done.

3. **Template URL hardcoding is a pattern risk in Flask apps without a linting rule.** There is no automated check preventing hardcoded URL paths in templates, making routing regressions invisible until deployment context changes.

4. **Mobile navigation conditional logic was not tested against the full set of `page_key` values.** The exclusion of `page_key == 'dashboard'` from bottom navigation was a code branch that was never exercised by a test asserting mobile nav presence across all operator pages.

5. **UX spec compliance for interactive log details (expand/collapse) was not expressed as a testable acceptance criterion.** The visual behavior was described in the UX design but did not surface as a machine-verifiable story outcome.

6. **Conversation context had no formal service contract in the codebase until Story 3.3 explicitly defined one.** The gap existed because FR9 rolling-window semantics were not mapped to a concrete implementation boundary in earlier epics.

---

## Concrete Process Improvements Identified

1. Require a full-lifecycle integration test for any wizard or multi-step flow (setup, escalation, operator navigation) before marking a story complete, not just unit acceptance tests.
2. Add a Flask template linting step or test assertion pattern that flags hardcoded URL paths and requires `url_for()` for all internal route references.
3. Assert mobile navigation presence for every `page_key` value in operator template tests so nav exclusion bugs are caught immediately.
4. Express interactive UX behaviors (expand/collapse, progressive disclosure) as testable acceptance criteria with explicit assertion patterns, not just visual descriptions.
5. Require a named seam (service module + interface contract) for any per-user stateful behavior before the story that introduces it begins implementation.

## Continuity Notes

- Epic 3 delivered the full operator control surface that was required before release-readiness work in Epic 4 could be meaningful. Despite Story 3.2 closing at review state, the escalation and setup contracts were coherent enough for downstream consumption.
- The most impactful lesson from this epic: operator-facing UX correctness (routing, accessibility, mobile nav) requires integration tests that simulate full page lifecycle states, not just unit coverage of individual helper functions.
- Agent selection atomic persistence and escalation reason taxonomy were the two structural decisions that aged best — both were consumed without change across five subsequent stories in Epics 4, 5, and 6.
